Dreamcast Software Development Standards/Ver. 2.00EU

2. Controllers and Control Ports

With Dreamcast, the controller is not controlled by an ID. Instead, controller ownership information is returned together with the key (or button) signal, indicating whether it is from a keyboard (system), gun (system) or control (system).

By providing key (or button) control instead of control by ID, we are considering the compatibility of different types of controllers that may appear in the future, as long as the required operating buttons are supported in existing applications.

In the following, a compatible controller refers to one that has the necessary keys (or buttons) for application operation.

Note: Functions of the A/B and START buttons, and the ten digital keys, are the guaranteed minimum controller functions. The X/Y buttons of the standard controller, L/R trigger and the ten analogue keys are not guaranteed to be present.

Note: The minimum keys (or buttons) required to operate the application with the standard controller are stored in disc IP information as 16-bit data. (This includes A, B, X, Y, and START for soft reset.)
In addition, when key combinations differ from the above-mentioned required minimum keys because of changes in key configuration, the keys should be based on the contents of the initial settings.

Note: A hardware keyboard is not currently a supported controller (system), so there is no specification assigning, for example, the Z key as the standard controller A button, as it is in Saturn. Operations with a hardware keyboard are limited to key input, with other operations performed by other controllers connected separately.

Required: When an incompatible controller is inserted in the control port, the game shouldn't stop.

In past development standards, it was written that applications should not be allowed to "start" or operate from an incompatible controller, but from this version, that has been amended to ...if the application does not stop, operation is possible...

Along with the above change, in the application instruction manual, it should be mentioned that "operation with incompatible controllers is not guaranteed" .

Recommended: If operation with an incompatible controller is possible, buttons or functions on the controller must substitute missing keys (or buttons) to ensure correct operation.

2.1 Using a Control Port

Refer to A or B listed below for the appropriate type of application and jump to that section.

A: An application in which a single user uses a single controller. (2.1.1)

B: An application in which a single user uses multiple controllers. (2.1.2)
(e.g., one person uses a steering wheel controller and keyboard, etc.)

2.1.1 Single User, Single Controller Applications

Required: As a rule, Main Unit port A should be used for player 1, port B for player 2, and so forth. When starting a game, start-up and operation should begin with the first port having a compatible controller connected, from A to D.

Required: Ports exceeding the number of players should not be allowed to start or operate.

When using an expansion socket as explained in 15.2 Memory Card (Visual Memory) Selection, it doesn't matter if the ports being used exceed the number of players.

For example, if you save a game to a memory card in an application that can only be operated by 1 or 2 people/ports, you will not encounter a problem if you save the game even if you select an expansion socket that has a 4 person/port controller inserted in it.

Example 1. In a fighting game (normally for up to 2 players) intended for either one or two players, if standard controllers are connected to ports A, B and D, port A should be for player 1 to start and operate, and port B for player 2. Starting or playing from port D should not be possible.

Example 2. In a 4-player fighting game there are no extra ports, so start and play should be possible from all ports with compatible controllers.

2.1.2 Single-User, Multi-Controller Applications

Required: When starting a game, any of ports A through D to which a compatible controller is connected should be enabled for start-up and operation (at the same time).

Example: Internet Explorer assumes screen operation from the standard controller and character input from the keyboard. However, there is no regulation about fixed ports, such as the standard controller should be in Port A or the keyboard should be in Port D.

2.2 Checking Control Ports

Required: Even though a compatible controller is not connected to a port when power is turned on, if it is connected before starting a game, it must be properly recognised and enabled for start and operation.

Check the ports frequently in the Title Loop before starting a game.

Required: Applications that can be joined in mid-play should frequently check the ports during game play, and enable ports when compatible controllers connected.

2.3 Case of Controller Removed from Port

Recommended: When a controller is removed from an active port after an application has started, insert a pause and display the warning "the controller has been removed or a memory card is being recognised" on the screen except when it is extremely difficult to do so.

"When it is extremely difficult to do so" means cases in which this warning cannot be displayed, such as in applications that don't support pauses created with the 12. Pause Function, screens with specs that don't allow pauses described in 12.4 Pause Limitations, or when pauses cannot be made for reasons of technology or capacity.

Not only when a controller is removed, but also when a memory card is inserted while an application is executing, the same status occurs for a certain time (about 0.5 seconds to recognise, and up to about 3 seconds if the Visual Memory battery is depleted).

The application side cannot determine if a controller is disconnected or VMS inserted, so the above caution message avoids the user misunderstanding that the controller has malfunctioned. It also appears when a VMS is inserted, but the above delays do not occur when a VMS is removed.

2.4 When a Controller is Removed and then Reconnected to the Same Port

Required: If a controller is removed from a port that has been used and a compatible controller is reconnected to the same port, input is resumed from the time of connection.

Connecting a different controller is okay as long as it is compatible.

If a controller includes external memory, it must be recognised (refer to 15. Backup).

2.5 A Controller is Reconnected to a Different Port During the Game

Required: Input is not resumed in an application where single player uses a single controller (2.1.1).

Recommended: In the cases of 2.1.2 Single-User, Multi-Controller Applications, input should be resumed even on a different port as long as the controller is compatible.

2.6 Peripheral Support

Follow the 18. Development Standards for Peripherals when developing applications with supported peripherals.
